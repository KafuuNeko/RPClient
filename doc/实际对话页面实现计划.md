# 实际对话页面实现计划

本文档用于交给 AI 直接继续实现 RPClient 的真实对话页面逻辑。实现时必须遵循 `doc/MVI框架开发规范指南.md`，并参考：

- `doc/SillyTavern Prompts 结构与世界书触发机制说明.md`
- `doc/SillyTavern 总结机制说明.md`
- SillyTavern 官方宏文档：https://docs.sillytavern.app/usage/core-concepts/macros/
- SillyTavern 官方 Prompt 文档：https://docs.sillytavern.app/usage/prompts/

## 总目标

将当前 `feature/chat` 从演示数据改为完整可用的真实对话页面：

1. 从 Room 读取真实 session、character、messages、world book entries。
2. 支持用户输入、发送消息、调用当前选中的 LLM Provider。
3. 支持流式输出和非流式输出，由 `AppModel` 中的开关控制。
4. 支持停止流式生成，并保留已生成内容。
5. 支持在对话页修改当前 session 启用的世界书条目。
6. 支持标题栏菜单查看/编辑当前 `title`、`summarize`、`userNote`、`creatorNotes`。
7. 支持 Prompt 宏替换能力，例如 `{{user}}`、`{{char}}`、`{{history}}`、`{{summary}}` 等。
8. 支持 SillyTavern 风格的世界书触发与 prompt 构建。
9. 支持手动编辑总结，并预留自动总结能力。

## 实现约束

1. 不要在 Compose 中写业务逻辑、IO 或 prompt 拼装。
2. `feature/chat/ui` 只渲染 `ChatUiState` 并发送 `ChatUiIntent`。
3. `ChatViewModel` 负责 intent 调度、状态更新、调用 repository / manager。
4. Prompt 构建、宏替换、世界书触发、总结构建必须放入 `libs/` 下的可复用类。
5. 不要修改 `libs/core` 基类。
6. 避免继续使用 `libs/model/RpUiModels.kt` 中标注为测试用途的 UI model；Chat feature 内应建立自己的 model。
7. 所有长耗时任务使用协程，并支持取消。
8. 真实对话逻辑完成后，`ChatViewModel` 中不得保留 preview 数据。

## 当前项目状态摘要

已有能力：

- `ChatRepository` 已支持 session/message 基础 CRUD。
- `ChatRepository.createSession(...)` 已支持创建 session。
- `ChatRepository.getSessionLorebookEntryIds(...)` 和 `updateSessionLorebookEntryIds(...)` 已封装 `lorebookEntrySet` JSON id 列表。
- `LLMRepository` 已支持：
  - `generateWithSelectedProvider(request)`
  - `streamGenerateWithSelectedProvider(request)`
- `LLMClient` 已定义非流式与流式接口。
- OpenAI Compatible / Gemini / Anthropic adapters 已有流式实现。
- `AppModel` 已有：
  - `currentLLMProvider`
  - `mainPrompt`
  - `summarizePrompt`
- `ChatActivity` 已可接收：
  - `EXTRA_SESSION_ID`
  - `EXTRA_FIRST_MESSAGE`

当前缺口：

- `ChatViewModel` 仍使用 preview 数据。
- `ChatLayout` 输入框不可编辑。
- 发送、停止、重生成仍是 toast 占位。
- streaming 开关未持久化到 `AppModel`。
- prompt 构建、宏替换、世界书触发、总结逻辑尚未实现。
- 对话页菜单和 session 元信息编辑尚未实现。

## 阶段 3：ChatViewModel 初始化真实数据

### 目标

`ChatUiIntent.Init(sessionId, firstMessage)` 读取真实 session，并处理创建对话时传入的开场白。

### 修改文件

- `ChatViewModel.kt`
- `ChatUiIntent.kt`
- `ChatUiState.kt`

### 任务

1. `ChatViewModel` 实现 `KoinComponent`，注入：
   - `ChatRepository`
   - `CharacterRepository`
   - `LorebookRepository`
   - `LLMRepository`
2. `onInit`：
   - 校验 `sessionId.toLongOrNull()`。
   - 读取 session，不存在则 toast 并 finish。
   - 读取 character。
   - 读取 messages。
   - 如果 messages 为空且 `firstMessage != null`：
     - 保存一条 `ChatMessage.Source.Char`。
     - 再刷新 messages。
   - 读取所有 lorebooks + entries。
   - 根据 `ChatRepository.getSessionLorebookEntryIds(session)` 标记 enabled。
3. `ChatMessage.Source` 映射：
   - `User` -> UI User
   - `Char` -> UI Assistant
   - `System` -> UI Narrator
4. 时间展示可先使用 `HH:mm`。

### 注意

如果 `firstMessage == null`，不能创建空消息。

### 验收

- 从主页点击已有 session 可看到真实消息。
- 从创建页进入新 session 时，如果传入开场白则自动出现第一条角色消息。
- `ChatViewModel` 无 preview 方法。

### 目标

### 修改 `ChatUiIntent`


### ViewModel 发送流程

`onSendMessage`：

1. 获取 `ChatUiState.Normal`。
2. 如果 `inputDraft.trim().isBlank()` 直接 return。
3. 如果正在生成，return 或提示。
4. 保存用户消息：

```kotlin
val userMessageId = mChatRepository.createMessage(
    sessionId = sessionId,
    source = ChatMessage.Source.User,
    content = input.trim()
)
```

5. 清空输入框并刷新消息列表。
6. 构建 prompt request。
7. 根据 `AppModel.streamEnabled` 选择非流式或流式生成。

### 非流式生成

1. 调用 `mLLMRepository.generateWithSelectedProvider(request)`。
2. 保存角色回复：

```kotlin
mChatRepository.createMessage(
    sessionId = sessionId,
    source = ChatMessage.Source.Char,
    content = response.content
)
```

3. 刷新消息。

### 验收

- 用户可输入。
- 点击发送后用户消息落库。
- 非流式模式下可以得到模型回复并落库。

## 阶段 5：流式输出与停止生成

### 目标

支持实时增量显示，用户可停止。

### 实现策略

1. 使用 `enqueueAsyncTask {}` 包裹生成任务。
2. `StopGeneration` 调用 `cancelActiveTaskAndRestore()` 或维护生成 Job 并取消。
3. 流式开始时：
   - 可先插入一条空的 `ChatMessage.Source.Char`，记录 `assistantMessageId`。
   - UI 中显示 `ChatGenerationState.Streaming(messageId, content)`.
4. 收到 `LLMStreamEvent.Delta`：
   - 拼接 `contentBuffer`。
   - 更新 UiState。
   - 可每 500ms 节流更新 DB，或停止/完成时再更新 DB。
5. 收到 Finished：
   - 将完整 content 保存到 assistant message。
   - 刷新消息。
6. 被 StopGeneration 取消：
   - 如果已有 partial content，保存 partial。
   - 状态回到 Idle。

### 注意

`LLMStreamEvent.Finished` 不一定所有 adapter 都稳定返回，收集 Flow 正常结束也应视为完成。

### 验收

- Streaming 开启时，UI 实时出现增量文本。
- Stop 后保留已生成文本。
- Streaming 关闭时仍走非流式。

## 阶段 8：世界书触发与注入

### 目标

实现接近 SillyTavern 的世界书机制：不是把所有启用条目都塞入 prompt，而是从候选条目中扫描触发。

### 当前数据限制

`LorebookEntry` 当前字段：

- `id`
- `lorebookId`
- `name`
- `keywords`
- `secondaryKeywords`
- `order`
- `depth`
- `category`
- `content`

没有字段：

- enabled
- constant
- probability
- insertion position
- budget
- recursive
- cooldown/sticky

因此初版按现有字段实现。

### 候选条目

候选条目 = 当前 session 的 `lorebookEntrySet` 中启用的条目。

### Scan Buffer

按文档，世界书扫描不应扫描最终 prompt 全部内容。初版 scan buffer：

1. 最近 N 条聊天消息，N 初版写死 4，后续可入 AppModel。
2. 当前用户刚输入的消息。
3. `session.userNote` 可选加入；建议初版不加入，避免长期稳定触发。
4. `session.summarize` 初版不加入，避免旧 key 反复触发。

### 触发规则

1. 如果 primary keywords 为空，不触发。
2. 任一 primary keyword 出现在 scan buffer 中则命中。
3. 如果 secondary keywords 非空，则至少一个 secondary keyword 也要出现。
4. 大小写忽略。
5. 命中后按 `order ASC, id ASC` 排序。

### 注入规则

初版：

- `depth <= 0`：作为一条 system message，放在 history 前。
- `depth > 0`：暂时仍作为 system message 放在 history 前，但内容前加标识：

```text
[World Info / Depth {{depth}}]
...
```

第二阶段再实现真正 In-Chat Depth 插入。

### 验收

- 对话页开关世界书条目后，生成 prompt 的候选条目变化。
- 只有 keyword 命中的条目进入 prompt。
- 条目按 order 排序。

## 阶段 11：总结机制

### 目标

实现手动编辑总结，并预留自动总结。

### 手动总结

阶段 10 已覆盖编辑 `summarize`。

### 自动总结任务

新增 `SummaryPromptBuilder`，输入：

- existing summary
- unsummarized messages
- char name
- user name
- words limit

输出 `LLMGenerationRequest`。

### 总结 prompt

如果 `AppModel.summarizePrompt` 为空，使用默认：

```text
请将以下聊天记录整理为一份简洁的剧情记忆摘要。

规则：
- 不要继续角色扮演。
- 不要替 {{char}} 或 {{user}} 生成新回复。
- 只总结聊天中已经实际发生的内容。
- 保留重要事实、关系变化、承诺、伤势、地点、目标、未解决冲突和当前场景状态。
- 排除推测、泛世界观设定、重复对白和无关细节。
- 控制在 {{words}} 字以内。

已有摘要：
{{summary}}

需要总结的聊天记录：
{{history}}
```

### 重要原则

根据文档，总结请求不应提交完整角色扮演 prompt 栈：

- 不提交世界书。
- 不提交角色卡。
- 不提交 creatorNotes。
- 不提交 userNote。
- 只提交已有 summary + 需要总结的历史。

### 保存

调用：

```kotlin
mChatRepository.saveSessionSummarize(
    sessionId = sessionId,
    summarize = newSummary,
    summarizedMessageIds = ids
)
```

### 自动触发策略

初版不强制启用。若 `AppModel.autoSummaryEnabled == true`：

1. 每次 assistant 回复完成后检查未总结消息数量。
2. 超过 `summaryTriggerMessageCount` 时执行总结。
3. 总结失败只 toast，不影响对话。

### 验收

- 手动编辑 summary 后 prompt 使用新 summary。
- 自动总结开启时可更新 `ChatSession.summarize` 并标记消息已总结。

## 阶段 12：Prompt 裁剪与上下文预算

### 目标

防止 prompt 超过 provider contextTokens。

### 初版策略

因为项目目前没有 tokenizer，可先使用字符数估算：

```text
estimatedTokens = text.length / 3
```

上下文预算：

```text
maxPromptTokens = provider.contextTokens - provider.maxTokens
```

裁剪顺序：

1. 保留 main prompt。
2. 保留角色核心定义。
3. 保留 summary。
4. 保留当前用户消息。
5. 保留最近历史，从新到旧加入。
6. 世界书按 order 加入，预算不足时跳过低优先级条目。

### 第二阶段

引入真实 tokenizer 或 provider 近似 tokenizer。

### 验收

- 长对话下不会无限提交全部历史。
- 最近消息优先保留。

## 关键实现细节

### ChatMessage.Source 映射

```kotlin
ChatMessage.Source.User -> LLMMessageRole.User
ChatMessage.Source.Char -> LLMMessageRole.Assistant
ChatMessage.Source.System -> LLMMessageRole.System
```

### 历史格式化

用于 `{{history}}`：

```text
{{user}}: ...
{{char}}: ...
System: ...
```

用于 Chat Completion messages：

- User 消息保持 role user。
- Char 消息保持 role assistant。
- System 消息可作为 system 或合并进 system context。

### First Message

创建页已传 `EXTRA_FIRST_MESSAGE`：

- 如果 session 无消息且 firstMessage 非空，进入 Chat 时保存为第一条 Char 消息。
- 如果 firstMessage 为 null，不保存。

### Regenerate

当前可暂缓。后续实现：

1. 找到最后一条 Char 消息。
2. 删除或标记旧回复。
3. 使用之前的 history 重新生成。
4. branch/swipe 模型当前数据库不支持，先做简单 regenerate。

## 最终验收标准

1. 新建 session 后进入 Chat 页面，能显示真实 session 与角色。
2. 如果有开场白，第一条角色消息自动落库显示。
3. 用户可以输入并发送消息。
4. 非流式模式可以生成完整回复并保存。
5. 流式模式可以实时显示增量回复。
6. 用户可以停止流式生成，partial 内容保留。
7. 世界书条目开关可持久化，并影响下一轮 prompt。
8. title / summary / userNote / creatorNotes 可编辑并持久化。
9. prompt 中 `{{user}}`、`{{char}}`、`{{history}}`、`{{summary}}` 等基础宏会替换。
10. 总结逻辑不混入世界书和角色卡。
11. `./gradlew.bat :app:assembleDebug` 通过。
12. `ChatViewModel` 不再包含 preview 数据。
